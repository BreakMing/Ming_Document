## 软件包的来历

像windows里的.exe后缀的安装包以及手机上.apk后缀的安装包，其本质都是软件包，我们在命令行中下载的软件包其本质和从软件商店以及官网下载都是一样的。

在以前开源软件只提供了源码文件，用户必须自己编译每个想在自己系统上运行的软件，这中间就会牵扯到手动编译、修改系统配置、补全所需依赖等各种问题，一般的用户根本搞不定。

这就极大的增加了软件使用的门槛，但这一套流程其实都是固定化的，所以开发者在完成开发后，把所有文件打包成一个标准文件，不就可以更方便大家的使用，大大降低了门槛。

那么在我们日常生活中，电脑和手机都可以运行QQ、微信等各种既能在电脑上运行，也能在手机上运行的软件，那么它们除了后缀，本质上有什么不同呢，

这个就牵扯到不同的系统、不同的编程语言的问题了，比如系统级的.deb（Debian/Ubuntu）、.rpm（CentOS/Fedora）；编程语言级的.whl（Python）。

## 包管理器

包管理器就像是手机上的应用商店一样，当我们需要往电脑上安装软件的时候就可以从它那里去下载我们需要的应用。

包管理器的作用不光是帮你下载软件，它是你的软件管家，它能够帮你找软件、下载软件安装包、安装软件、更新和卸载，这些你都不用操心，它一个人就能帮你完成。

而包管理器也可以分为**系统级的包管理器**和**编程语言的包管理器**。

像我们使用的**apt、winget、brew**这样的命令都可以理解为是系统级的包管理器

而像**pip、npm**这些都是编程语言的包管理器

系统级的包管理器可以想象成日常生活中的日用品，像拖把、桌子、床什么的它们是在什么地方都可以用的

而编程语言的包管理器可以想象成你在写作业时要用水笔、橡皮、尺子这些、这些工具是专用的，它是针对于这个事物专用的工具。

### 下载命令

这是一个宽泛的概念，包管理包含了下载这一功能，但我们使用一些工具也可以来下载我们想要的东西，比如说crul和wget，

我们先来实际操作下看都出现了什么现象

我们先执行下面这段命令

```
curl https://www.baidu.com
```

它给我生成了这样一段东西，如下如图所示：

![](assets/Package%20manager/file-20260320171308149.png)

我们先不管curl具体在做什么，我们再看看wget会怎么做，一样输入下面这段命令：

```
wget http://www.baidu.com
```

![](assets/Package%20manager/file-20260320171609585.png)

我们发现它确实访问了百度官网，但在我本地保存了一个index.html文件，那这个文件中有什么呢，我们用下面这个命令来看一下。

```
cat index.html
```

![](assets/Package%20manager/file-20260320171839802.png)

有没有发现它所保存的内容与curl命令输出的内容是一样的。

其实这两个命令都可以去访问网址，但不同的是，crul默认会把网址内容打印在终端，而wget会把网址内容下载并保存成文件。

如果只是下载软件或者下载资料什么的，使用wget会更加方便简单，它没有复杂的参数，对文件下载还有优化。

而curl像一个多面手，它什么都能做，能支持很多网络请求，以及网络协议，也可以下载文件。

### 版本冲突

假设我们电脑上有多个项目要执行，但不同项目间可能是基于库的不同版本来写的，如果我们统一只有一个版本就会导致，有的项目可以使用这个版本的库，而有的项目会因为库版本的不同，导致用法或内容上产生了改变，导致项目不能执行
